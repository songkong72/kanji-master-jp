# 한자 데이터 품질 지침 (Kanji Data Quality Guidelines)

한자 데이터(`kanjiData.json`)를 추가하거나 수정할 때 반드시 아래 규칙을 준수해야 합니다.

## 🔴 필수 규칙 (MUST)

### 1. 예문(on_sentence, kun_sentence)은 반드시 일본어로 작성
- **예문의 `text` 필드는 반드시 일본어(일본어 한자 + 히라가나/카타카나)로만 작성해야 합니다.**
- 한글이 절대 섞이면 안 됩니다.
- **`examples[].reading` 필드 및 루비 표기의 `()` 내부**에도 한글이 절대 포함되면 안 됩니다.
- **`mean` 필드만 한국어 번역**으로 작성합니다.

#### 1-1. 실생활 밀착형 평이한 예문 작성 원칙
- **난이도 조절**: 지나치게 학술적이거나 기술적인 문장(예: "소망을 투사하다", "소수점 이하 반올림")보다는 **일상생활에서 매일 사용되는 평이하고 쉬운 문장**을 우선적으로 작성합니다.
- **상용 핵심 어휘 중심**: 학습자가 바로 써먹을 수 있는 상용 어휘(예: "쓰레기를 버리다", "사과를 먹다", "학교에 가다")를 활용하여 예문을 구성합니다.
- **문장 길이**: 너무 길지 않고 간결한 호흡의 문장을 지향합니다.

#### ✅ 올바른 예시
```json
"on_sentence": {
    "text": "[虹](こう)가[空](そら)에[架](か)かった。", // ❌ 에(X) -> に(O) - 아래가 정답
    "text": "[虹](こう)が[空](そら)に[架](か)かった。",
    "mean": "무지개가 하늘에 걸렸다."
}
```

### 2. 예문 및 단어는 다양한 발음과 활용형을 반영 (핵심 지침)

#### 2-1. 1발음 1예문 원칙 (Multi-Reading Handling)
- 훈독(kun_reading)이나 음독(on_reading)에 **서로 다른 의미나 발음법이 2개 이상(예: ゆび, さす) 기재된 경우**, 학습자가 각각의 쓰임새를 모두 익힐 수 있도록 **각 발음별 예문을 모두 제공**해야 합니다.
- 이 경우 `on_sentence`나 `kun_sentence` 필드는 객체가 아닌 **배열(Array) 형태**로 작성합니다.
- **발음 일치 주의**: `on_sentence`에는 반드시 음독(on_reading) 발음이 포함된 예문을, `kun_sentence`에는 반드시 훈독(kun_reading) 발음이 포함된 예문을 작성해야 합니다. (음독 섹션에 훈독 단어 사용 금지)

#### 2-2. 실전 활용 단어(`examples`)의 다양성
- 음독과 훈독이 모두 존재하는 경우, **대표적인 음독 단어 1개와 대표적인 훈독 단어 1개**를 섞어서 제공하는 것을 원칙으로 합니다.
- **라벨링 필구**: `mean` 필드 끝에 반드시 `(음독)` 또는 `(훈독)` 라벨을 추가하여 발음의 성격을 명시합니다.
- **특수 발음 예외**: 음/훈독 이외의 특수 발음(아테지, 숙자음 등)이라도 실생활 사용 빈도가 매우 높다면(예: 今日, 大人) 예외적으로 포함합니다.

#### 2-4. 복합어(Compound Words) 사용 및 1:1 발음 매칭 원칙 (엄격 준수)
- **단일 한자 금지**: 실전 활용 단어(`examples`) 리스트에는 **학습 중인 한자 1개만 있는 단어(Single Kanji)를 절대 사용하지 않습니다.** 반드시 2개 이상의 글자가 결합된 **복합어(예: 攻 -> 攻撃, 攻める)**를 선정해야 합니다.
- **1:1 발음 매치**: 한자에 음독이 1개, 훈독이 1개 있다면 단어 리스트에도 반드시 **음독 단어 1개, 훈독 단어 1개** 이상이 포함되어야 합니다. 지침에 명시된 모든 주요 발음에 대해 최소 1개 이상의 대표 단어를 제공해야 합니다.
- **훈독 누락 금지**: 훈독 발음이 존재하는 경우, 반드시 훈독을 활용한 단어(주로 동사나 형용사 활용형)를 리스트에 포함시켜야 합니다.

#### 2-5. 단어(examples) 정렬 순서 원칙 (엄격 준수)
- 실전 활용 단어 배열은 항상 일정한 순서로 정렬해야 합니다: **음독 > 훈독 > 훈독/특수** 순으로 배치합니다.
- `mean` 필드에 명시된 라벨 `(음독)`, `(훈독)`, `(훈독/특수)` 등을 기준으로 정렬하며, 이 순서가 어긋날 경우 데이터 불량으로 간주합니다.
- 예: `(음독)` 단어들이 먼저 나오고, 그 뒤에 `(훈독)` 단어, 마지막에 `(훈독/특수)` 또는 `(숙자훈)` 단어가 위치해야 합니다.


#### ✅ 다중 예문 적용 예시 (JSON)
```json
"kanji": "指",
"kun_reading": "ゆび/さ.す",
"kun_sentence": [
    { "text": "[指](ゆび)を[怪](け)[我](が)する。", "mean": "손가락을 다치다." },
    { "text": "[時](と)[計](けい)の[針](はり)이 12[時](じ)を[指](さ)す。", "mean": "시곗바늘이 12시를 가리키다." }
]
```


### 2-3. 의미에 따른 정확한 읽기(히라가나) 매칭
- 동일한 한자 단어라도 **의미(뜻)에 따라 히라가나 읽기가 달라지는 경우**가 있습니다. 한국어 단어 뜻(`mean`)과 일본어 읽기(`reading`)가 서로 모순되지 않고 정확히 매칭되는지 확인해야 합니다.
- 예시: `一日`의 경우
  - 의미가 **"하루/1일"**일 때는 `いちにち` (음독)
  - 의미가 **"초하루(매월 1일)"**일 때는 `ついたち` (훈독/특수)
- 따라서 `word`, `reading`, `mean` 세 가지 필드가 문맥상 일치하도록 신중하게 기재해야 합니다.
- **특수 발음(숙자훈) 라벨링 규칙**: `ついたち`와 같이 한자의 음가와 상관없이 단어 전체에 순수 고유어 발음이 매칭되는 **숙자훈(熟字訓)**의 경우, 학습자에게 예외 규칙임을 명확히 알리기 위해 반드시 **`(훈독/특수)`** 또는 **`(숙자훈)`**이라고 병기해야 합니다.

### 3. 루비 표기(후리가나) 형식
- 한자에 읽기를 붙일 때는 `[漢字](よみがな)` 형식을 사용합니다.
- 복합어 사용 시 한글자 단위가 아니라 **단어 전체를 묶어 후리가나(발음기호)를 표기**하는 것을 원칙으로 합니다. (예: `[新緑](しんりょく)`)
- **부분 후리가나 절대 금지**: 단어에서 학습 중인 한자만 따로 빼서 발음기호를 다는 것은 지양하며, 단어 전체에 통합하여 달아야 합니다.
- **타겟 한자 및 루비 자동 강조 (스마트 렌더링)**: 후리가나가 그룹으로(단어 전체로) 묶여 있더라도, UI 렌더링 엔진(`renderRuby`)이 자동으로 **학습 중인 한자(Target Kanji)**와 **그에 해당하는 발음 조각**만 정확히 분리하여 강조색을 입힙니다.
  - **오쿠리가나(보내기가나) 무결성**: 훈독에 점(.)이 포함되어 있어도 어간과 전체 발음을 모두 유연하게 인식합니다. (예: `償(つぐな.う)`인 경우, 예문의 `つぐなう`나 `つぐな` 모두 완벽하게 매칭되어 색상이 칠해집니다.)
  - **음운 변동(촉음/연탁) 자동 지원**: 단어 구성 시 발생하는 발음 변화(예: `けつ` -> `けっ`, `は` -> `ば` 등)도 시스템이 배후에서 자동으로 추론하여 매칭합니다. 따라서 **발음 변동을 이유로 후리가나 태그를 인위적으로 쪼갤 필요가 전혀 없습니다.**

### 4. 데이터 구조 (필드 정의)
```json
{
    "id": 1,
    "kanji": "漢",
    "meaning": "한나라 한",
    "on_reading": "カン",
    "kun_reading": "-",
    "category": "N3",
    "explanation": "한국어 암기 팁",
    "examples": [
        { "word": "漢字", "reading": "かんじ", "mean": "한자" }
    ],
    "on_sentence": { "text": "...", "mean": "..." }, // 또는 배열 []
    "kun_sentence": { "text": "...", "mean": "..." }  // 또는 배열 []
}
```

### 5. examples의 mean은 한국어, word는 일본어
- `examples[].word`: 일본어 한자/단어
- `examples[].mean`: 한국어 뜻
- `examples[].reading`: 히라가나 읽기 (한글 절대 금지)

## 📋 체크리스트
- [ ] 훈독/음독이 2개 이상인가? -> 예문을 **배열**로 작성했는가?
- [ ] 단어(`examples`)에 음독과 훈독 사례가 **골고루** 섞여 있는가?
- [ ] 단어(`examples`)의 뜻 필드에 **(음독)** 또는 **(훈독)** 라벨을 명시했는가?
- [ ] 루비 표기 내부에 **한글**이 섞여 있지 않은가?
- [ ] 복합어 루비가 **단어 전체 단위**로 묶여 있는가? (`[日](に)[本](ほん)` X -> `[日本](にほん)` O)

### 6. 섹션별 하이라이트 색상 일치
- **음독 예문 하이라이트**: 한자의 강조 색상은 음독 테마 색상(적색 계열)과 일치해야 합니다. (renderRuby 호출 시 타입 지정)
- **훈독 예문 하이라이트**: 한자의 강조 색상은 훈독 테마 색상(녹색 계열)과 일치해야 합니다.
- 이는 예문의 성격에 따라 학습자가 시각적으로 음/훈독을 즉시 구분할 수 있게 하기 위함입니다.

### 7. 훈독 오쿠리가나 점(.) 표기 처리 (UI 렌더링)
- 데이터베이스(`kanjiData.json`)의 `kun_reading` 필드에는 한자 부분과 오쿠리가나(보내기 가나) 부분을 구분하기 위해 점(.) 기호가 포함되어 있습니다. (예: `ひと.つ`)
- **UI 화면(렌더링) 규칙**: 사용자 화면에 훈독을 노출할 때는 이 점(.)이 문자로 직접 보이게 해서는 안 됩니다.
- 텍스트 렌더링 시 점(.)을 파싱하여, 앞부분(한자 영역)과 뒷부분(오쿠리가나 영역)을 시각적으로만 구분(`style` 분리 등)하고, 점(.) 문자는 숨기거나 제거하여 자연스러운 단어(`ひとつ`)로 읽히도록 처리해야 합니다.
